# InnoNetwork 4.1.0 Release Notes

InnoNetwork 4.1.0 is a hardening release. Every change is API-additive
or documentation-only — existing 4.0.x consumers can adopt 4.1.0 without
any source change. The release focuses on fixing latent correctness
bugs, tightening streaming back-pressure, sharpening macro diagnostics,
and bringing public docstrings in sync with runtime behaviour.

## Highlights

- **Circuit breaker correctness**: a 4xx response no longer resets the
  consecutive-failure window, so transport failures interleaved with
  application-level 4xx errors still trip the breaker on schedule.
- **Configurable WebSocket close handshake**: the previously hard-coded
  3-second wait that follows `cancel(with:reason:)` is now exposed as
  `WebSocketConfiguration.closeHandshakeTimeout: Duration` (default
  `.seconds(3)`, negative values clamp to `.zero`).
- **Explicit streaming buffering policies**: every `AsyncStream` and
  `AsyncThrowingStream` factory owned by InnoNetwork now declares a
  `bufferingPolicy`. Streaming responses, download delegate events, and
  event-hub consumer streams use `.unbounded` (no event loss); the
  `NetworkMonitor` path stream uses `.bufferingNewest(16)` so a slow
  observer only ever sees the most recent network state.
- **Macro diagnostics with source locations**: `@APIDefinition` and
  `#endpoint` errors now point at the offending argument, label, or
  placeholder rather than dropping every diagnostic on the macro
  attribute as a whole. IDE underlines now match the actual mistake.
- **Documentation sync**: `WebSocketState.connecting.nextStates` lists
  the `.reconnecting` transition the manager already drives,
  `RequestCoalescingPolicy` documents `Authorization` handling in the
  dedup key, and `NetworkEventHub.publish` documents the partition
  close lifecycle.

## Fixed

- **CircuitBreaker (P1.1)**: `recordStatus` previously reset the
  consecutive-failure window on any non-5xx response, including 4xx.
  A 4xx is a semantic failure — the transport itself succeeded — so it
  must not undo accumulated transport failures. The handler is now
  state-aware: 5xx always counts as a transport failure; in `.closed`
  a 4xx is a no-op (so it cannot mask accumulated transport failures);
  in `.halfOpen` a 4xx releases the probe slot and closes the circuit
  because the probe purpose is to confirm the transport works, and a
  4xx response satisfies that contract; 2xx/3xx clears the failure
  window in every state. `recordSuccess` has no production callers,
  so the close path runs entirely through `recordStatus`. The window
  guarantee is now: any number of 4xx responses interleaved with
  `policy.failureThreshold` consecutive 5xx or transport failures
  will still open the circuit, and a half-open probe that returns any
  status (2xx/3xx/4xx) reopens admission instead of wedging on
  `probeInFlight`.
- **Streaming back-pressure (P1.7)**: every async stream factory now
  declares a `bufferingPolicy` explicitly. Streaming response bytes,
  download delegate events, and event-hub consumer streams use
  `.unbounded` (event loss would corrupt task lifecycles or drop
  server-emitted records); `NetworkMonitor` snapshots use
  `.bufferingNewest(16)` because only the latest path state is
  meaningful.
- **WebSocket lifecycle table (P2.8)**:
  `WebSocketState.connecting.nextStates` now includes `.reconnecting`.
  The manager already promotes a connecting task to `.reconnecting`
  whenever a handshake error is classified as `transportFailure` and
  the reconnect coordinator returns `.retry`, so this only repairs the
  documentation surface used by consumers introspecting the table.
- **Macro diagnostics (P1.14)**: `InnoNetworkMacros` diagnostics now
  carry `SourceLocation` information so IDEs can underline the exact
  offending argument or placeholder. `MacroDiagnostic` exposes a new
  `error(at: some SyntaxProtocol)` helper that wraps the diagnostic in
  a `DiagnosticsError` anchored at the supplied node;
  `APIDefinitionMacro` and `EndpointMacro` thread the most precise
  syntax node through every throw site.
- **EventHub late-publish semantics (P1.3, doc only)**:
  `NetworkEventHub.publish` now states that observers are bound at
  publish time (the hub is not a replayable subscriber stream) and
  that publishes after `finish(requestID:)` are intentionally dropped
  because the consumer side of the partition has already torn down.

## Added

- **`WebSocketConfiguration.closeHandshakeTimeout: Duration`**: the
  manager waits at most this duration for the close handshake to
  finish after a client-initiated `cancel(with:reason:)` before
  finalising the disconnect locally. The default (`.seconds(3)`)
  matches the previous hard-coded value, so existing code is
  unaffected. Negative values clamp to `.zero`. `Presets.safeDefaults`
  and `Presets.advancedTuning` thread the new field through their
  builders.

## Documentation

- **`RequestCoalescingPolicy`** now documents the interaction between
  coalescing and `Authorization`: the header participates in the
  dedup key by default, so callers presenting different tokens never
  share a transport, and `RefreshTokenPolicy` is the supported way to
  recover token-mismatch peers individually. Opting into
  Authorization-agnostic dedup remains possible but is called out as
  only safe when every caller in the cohort presents identical
  credentials.
- **`NetworkEventHub.publish`** documents the publish-time observer
  binding and the closed-partition drop rule (see `Fixed` above).

## Migration

No migration is required. Every change in 4.1.0 is additive or
documentation-only:

- **CircuitBreaker (P1.1)** is a behaviour change in the strict sense,
  but only in the direction of "the breaker now opens when it was
  supposed to." Consumers that were relying on the old 4xx reset
  behaviour to keep a host marked healthy were already on an unstable
  contract — that path now matches the documentation.
- **`WebSocketConfiguration.closeHandshakeTimeout`** has a default that
  reproduces the previous hard-coded 3-second wait, so existing
  initialisers continue to compile and behave identically.
- **`bufferingPolicy`** is now declared at every stream factory site;
  observable behaviour matches the previous defaults for any consumer
  whose receiver keeps up with the producer.

## Known follow-ups

The following items from `docs/reviews/4.x-comprehensive-evaluation.md`
are intentionally not in 4.1.0 and are tracked for a follow-up release:

- **P1.2** Conditional revalidation `Vary` snapshot preservation when
  a 304 brings a different `Vary` header. Requires a
  `ResponseCachePolicy`-level decision (skip cache write versus
  re-key) that is too policy-shaped to ship in a hardening release.
- **P2.1** Refresh-aware coalescer dedup key. Concurrent callers that
  enter the coalescer with stale tokens during a refresh-in-flight
  window can still receive a stale 401 from the originator. The
  recovery path (`RefreshTokenPolicy`) already retries each waiter
  individually after the originating request returns 401, so the
  observable outcome is correct, but the policy surface change to
  segregate refresh-pending lanes is large enough to deserve a
  dedicated PR.
