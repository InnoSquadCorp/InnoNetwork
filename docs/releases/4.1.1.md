# InnoNetwork 4.1.1 Release Notes

InnoNetwork 4.1.1 closes the remaining P1 / P2 items from the 4.x
comprehensive evaluation and lands a batch of P3 cookbook
documentation. Every change is API-additive, documentation-only, or
test-only; the one runtime behaviour change (P1.2's 304-with-new-Vary
handling) is a correctness fix toward RFC 9111 §4.3.4 semantics. No
4.1.0 consumer needs source changes to adopt 4.1.1.

## Highlights

- **304-with-new-Vary preservation (P1.2)**: a conditional revalidation
  that succeeds (304 Not Modified) but carries a different `Vary`
  header than the cached entry no longer rewrites the entry under the
  new vary dimension. The executor refreshes the existing entry's
  `storedAt` while leaving its headers, body, and vary snapshot
  untouched.
- **Refresh-aware coalescer lanes (P2.1)**: concurrent callers
  entering ``RequestExecutor`` while a token refresh is in flight
  receive a unique coalescer-lane suffix so a peer's pre-refresh
  transport result cannot leak into another caller's resumed
  continuation.
- **URLError mapping contract (P1.11)**: the intentionally narrow
  `URLError` → `NetworkError.timeout` mapping (only `timedOut` and
  `cannotConnectToHost`) is now pinned by `NetworkErrorTimeoutTests`
  and documented on `NetworkError.mapTransportError(_:)`. DNS and
  reachability failures continue to surface as `.underlying`.
- **End-to-end POST 503 + Retry-After + Idempotency-Key coverage
  (P1.13)**: integration tests now tie Retry-After parsing, unsafe-
  method idempotency policy, and exponential-backoff fall-through
  together against the executor.
- **Migration cookbook off `DownloadManager.shared` (P2.3)**: a new
  `SharedManagerMigration` DocC article walks through the
  dependency-injected per-feature manager pattern for callers
  preparing for a future `@available(*, deprecated, ...)` removal of
  the singleton.
- **Cookbook quartet (P3)**: WebSocket subprotocol / heartbeat
  policy, foreground/background transition, vendor-neutral
  observability adapters, and an OpenAPI Generator workflow recipe
  ship as DocC articles.

## Fixed

- **Response cache (P1.2)**: a 304 response carrying a `Vary` header
  different from the one snapshotted with the cached entry no longer
  rekeys the stored entry. The new
  ``ResponseCachePolicy/notModifiesRevisesVary(cached:notModifiedHeaders:)``
  helper detects same-Vary, normalized-token, different-Vary,
  added-Vary, and no-Vary-on-304 cases. Behaviour is unchanged when
  the 304 carries the same `Vary` header.
- **Refresh-aware coalescer lanes (P2.1)**: the request executor now
  reads ``RefreshTokenCoordinator/isRefreshInProgress`` (newly
  package-visible) before composing the coalescer dedup key. If a
  refresh is in flight, the dedup key carries a per-caller UUID lane
  suffix so concurrent callers see distinct transports during the
  refresh window. Once refresh completes, dedup behaviour returns to
  the steady state.

## Added

- ``RefreshTokenCoordinator/isRefreshInProgress`` — package-scoped
  point-in-time observation of refresh-coordinator state, used by
  ``RequestExecutor`` for lane segregation.
- ``RequestDedupKey`` gains an optional `refreshLane: UUID?`
  initializer parameter so the executor can synthesize per-caller
  dedup keys during refresh-in-flight windows without changing the
  default coalescing surface.

## Tests

- `ResponseCacheVaryTests` covers the new 304-with-new-Vary helper
  across same-Vary, normalized-token, different-Vary, added-Vary,
  and no-Vary-on-304 scenarios.
- `ResiliencePolicyTests.etagNotModifiedWithChangedVaryPreservesSnapshot`
  exercises the executor's 304-with-new-Vary path end-to-end.
- `DownloadTaskStateTransitionTests` (P1.10) backfills regression
  coverage for `DownloadTask.updateState`'s lifecycle guard.
- `NetworkErrorTimeoutTests` (P1.11) adds a contract-lock group
  pinning every documented `URLError` code mapping plus the preserved
  underlying URLError code.
- `RefreshCoalescerRaceTests` (P2.1) covers two concurrent OLD-token
  callers entering during a held refresh, validates that both
  callers retry with the new token, and that
  `RequestDedupKey`'s `refreshLane` suffix distinguishes
  otherwise-identical keys.
- `IdempotencyRetryIntegrationTests` (P1.13) ties Retry-After
  parsing, unsafe-method idempotency, and exponential-backoff
  fall-through together end-to-end.
- `WebSocketLifecycleReducerFuzzTests` (P3) drives the reducer with
  five SplitMix64-seeded random walks and pins three pure-function
  invariants (determinism, generation monotonicity, stale-callback
  rejection) plus the manual-disconnect and max-reconnect-exceeded
  absorbing terminals.

## Migration

No migration required. Every change in 4.1.1 is additive,
documentation-only, or test-only:

- **P1.2** changes the 304 cache-write path to refresh-and-preserve
  rather than rekey. Callers that depended on the previous rekey
  behaviour (which deviated from RFC 9111 §4.3.4) were on an unstable
  contract; the new behaviour is the documented one.
- **P2.1** changes coalescer key composition only when refresh is
  in flight. Outside of refresh windows, dedup behaviour is
  unchanged.
- The new public surfaces (``RefreshTokenCoordinator/isRefreshInProgress``,
  ``RequestDedupKey`` `refreshLane:` parameter) are additive and
  package-scoped or default-nil.

## Known follow-ups

The following P3 items remain candidates for a future 5.0 major
release rather than a 4.1.x point release because they require
breaking refactors:

- **StateMachine / Reducer protocol unification** across Download,
  Refresh, and WebSocket lifecycles.
- **Lifecycle reducer generalization** for Download and Refresh,
  matching the WebSocket reducer model.
- **Persistent Response Cache companion product** implementation
  (the policies are documented in the new RFC).
- **Public Execution Policy protocol** and an NIO transport adapter.
- **WebSocket `permessage-deflate` (RFC 7692)** — requires an NIO
  transport.

Smaller P3 nitpicks (`URLQueryEncoder` array convention options,
`WebSocketManager.shared` deprecation, `AppendLog` snapshot/compaction
RFC) remain tracked in
`docs/reviews/4.x-comprehensive-evaluation.md` §1.5.
